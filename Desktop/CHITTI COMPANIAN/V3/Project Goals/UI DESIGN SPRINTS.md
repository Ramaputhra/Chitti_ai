This is exactly where I want to protect the project from derailing again.

**I would NOT sprint by screens.**
I would sprint by **user experience**.

For example, don't have a sprint called "Settings UI." Instead, have "User can control CHITTI." The UI is just the presentation of that capability.

### Why this order?

Notice that **every sprint delivers something Boss can immediately use**. We never build a "UI framework sprint" or a "widget library sprint." Each sprint ends with a demonstrable experience, while the underlying architecture remains untouched. That's the same product-first discipline that got the project back on track after the earlier derailment.
